    Solution

    Insertion Anomaly

    Inserting ProvNo and ProvSpeciality without any corresponding patient details is not possible.

    Deletion Anomaly

    When a patient record has to be deleted multiple records may have to be deleted which may result in anomaly.

    Updation Anomaly

    Updating the City of same patient in one place while leaving it unchanged at other place may result the inconsistent database,because same patient details may be stored in multiple places for cases of multiple visits.

    The functional dependencies in the given relation are:

    VisitNo -> VisitDate //Partial dependency

    VisitNo -> PatNo. //Partial dependency

    VisitNo -> PatAge //Partial dependency

    VisitNo -> PatCity //Partial dependency

    VisitNo -> PatZip //Partial dependency

    PatNo -> PatAge //Transitive Dependency

    PatNo -> PatCity //Transitive Dependency

    PatNo -> PatZip    //Transitive Dependency

    ProvNo -> ProvSpeciality    //Partial dependency

    VisitNo, ProvNo -> Diagnosis //Full dependency

    The relation is not in BCNF.

    Step 1: The relation is decomposed to 2NF by removing all the partial dependencies as shown below:

    Relation R1

    VisitNo ProvNo Diagnosis

    Primary Key:  VisitNo, ProvNo

    Relation R2

    VisitNo VisitDate PatNo PatAge PatCity PatZip

    Primary key: VisitNo

    Relation R3

    ProvNo ProvSpeciality

    Primary key: ProvNo

    Step 2

    The transitive dependencies are eleminated from relation R2 as shown below:

    Relation R1

    VisitNo ProvNo Diagnosis

    Primary Key:  VisitNo, ProvNo

    Relation R21

    VisitNo VisitDate PatNo

    Primary key: VisitNo

    Relation R22

    PatNo PatAge PatCity PatZip

    Primary key: PatNo

    Relation R3

    ProvNo ProvSpeciality

    Primary key: ProvNo

    Now the above four decomposed relations are in BCNF
